SiardApi - Users' Manual - Purpose

Purpose of the Project

SiardApi was developed as part of the SIARD project. It is used by the SiardCmd implementation of the SIARD project in order to read and write SIARD files adhering to the standardized SIARD Format 2.1.

It is a product of the Swiss Federal Archives serving as a reference implementation for the new SIARD Format, which the Swiss Koordinationsstelle für die dauerhafte Archivierung elektronischer Unterlagen (KOST) uses to replace the previous standard version 1.0 eCH-0165.

Changes from the SIARD Format 1.0

Numerous changes were introduced in SIARD Format 2.1. They are documented in the standard document.

The main changes are:

  • SIARD Format 2.1 is based on SQL:2008, whereas version 1.0 was based on SQL:1999;
  • large objects (BLOBs, CLOBs, ...) can now be stored externally in the file system;
  • user-defined data types (UDT, DISTINCT) and arrays are supported;
  • SIARD Files may use compression.

Changes from the final draft of SIARD Format 2.0

The SIARD Format 2.1 is an upgrade from the final draft of the SIARD 2.0 format developed together with the E-ARK project. The version 2.0 of the SIARD Format is now deprecated because of a number of shortcomings which were addressed in version 2.1.

The main changes from this format (final draft 2.0) are:

  • Change of the folder name for easy version recognition (e.g. for Pronom) from version to siardversion to provide clarity, that it is a SIARD file.
  • A number of corrections in the metadata.xsd (e.g. the regular expressions for the predefined data types).
  • In accordance with standard ISO SQL usage the builtinType has been renamed to predefinedType in the metadata.xsd.
  • The predefined data types BINARY and VARBINARY have been added and the predefined type BIT was removed, because SQL:2008 is the basis for the SIARD Format 2.1 standard. The types BINARY and VARBINARY were not yet available in SQL:1999 but they were added in SQL:2008. The BIT type is not part of SQL:2008, as it can adequately be replaced by BOOLEAN (for BIT(1)) or BINARY.
  • The types primaryKeyType and candidateKeyType in the metadata.xsd were recognized as being identical. Therefore they were replaced by the single type uniqueKeyType for both primary and candidate keys.
  • The element folder has been removed from columnType and fieldType because it is superfluous.
  • An element name has been added to fieldType and parameterFieldType, because the elements of an ARRAY type must be represented by a field too. It must be possible to reference fields by name.
  • Elements defaultValue and nullable have been added to attributeType, because these properties are supported by JDBC and partly by SQL:2008. (Constraints on nullability are not really acceptable for attributes according to SQL:2008 but supported by JDBC. In SIARD Format 2.1 we support it because it may be supported by SQL in the future. But we recommend, not to use the nullable element for attributes.)
  • Element specificName has been added to routineType, because the name alone does not identify a possibly overloaded routine in the schema.
  • A generic header/table.xsd has been added to the SIARD file which serves as a template for the individual content/schema<n>/table<n>/table<n>.xsd files.
  • The following XSD schema URLs are used:
    http://www.bar.admin.ch/xmlns/siard/2/metadata.xsd and
    http://www.bar.admin.ch/xmlns/siard/2/table.xsd.
    The idea is, that minor, backward-compatible changes of the XSD can be published as a new release under the same version-URL with only the version attribute inside the XSD including the release number (recommendation in the excellent Definitive XML Schema by Priscilla Walmsley).
  • Element cardinality has been removed from typeType and added to columnType, attributeType and parameterType, because SQL:2008 does not support arrays as structured types but as anonymous collection types instead. For the same reason the value ARRAY has been removed from categoryType.
